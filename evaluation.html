Critical Evaluation

User Interface/User Experience

We recognised that a responsive and easy-to-use user interface is extremely important for efficient communication of MND patients

As a result, the UI is extremely simple and intuitive. All of the buttons are labelled and different functionalities are split into different columns. Additionally, the user interactions are very responsive. Once the user presses on the smart generate button or the manual generate button, it will display a scrolling wheel animation with text notifying the user that the audio generation is in progress.

After the audio has been generated, the UI will be immediately updated with the correct audio file to play.

Functionality

Our application covers 100% of the functional requirements of the MoSCoW list. Within our demonstration videos, we have demonstrated all functionalities to ensure that they are working perfectly as intended.

Stability

Our team followed rigorous testing procedures throughout our development cycle. To ensure the correctness of all functionalities, we implemented unit tests using the unittest framework. Furthermore, we implemented custom error handling to maintain the accuracy and reliability of important functions such as correct text and emotion being stored.

Efficiency

The speed of audio generation has been a key concern throughout the project cycle. To ensure efficient communications for MND patients, we need to make the audio generation process as fast as possible while maintaining a high quality of emotional intonations. This was a difficult task since it is usually a tradeoff between speed and quality of audio. Throughout our research and testing, while our audio generation model (Vall-E-X) performed worse than other models such as EmotiVoice, its quality was much better, as evidenced by the user acceptance testing. Additionally, by using more powerful hardwares such as GPUs, we can significantly improve the speed of audio generation for our application.

Compatibility

The application can be deployed for Windows and Linux. Although it is a web application, its current capabilities can only run on the local machine. However, if we deploy it to a cloud service such as Streamlit Cloud, our application can be used on different machines and operating systems whithout needing to set up a coding environment and installing dependencies.

Maintainability

The code base is well-documented and we have used the Model-View-Controller design pattern, which enhances application maintainability. This is because the separating of data, logic and presentation layers make code reusable and easier to modify without affecting other components.

Project Management

We approached the project with an agile methodology, building new prototypes over time. For our project management strategy, we implemented weekly meetings outside of our lab sessions, and pair programming sessions. We found pair programming sessions to be extremely effective, as it promotes collaboration and reduces many errors while improving code quality.

We also utilised the version control system Git to keep track of our code progression and enable all team members to work on the application together. For instance, while Andy was working on the Streamlit frontend, Maestro was writing Python scripts to generate emotional audio.

Future Work

Given more time, there are a number of things we would improve on in the project:

  1. Attempting other solution methods: Due to time as well as hardware constraints, we were unable to try implementing some of the more complex solutions to this project such as further training sentiment analysis and audio generator models. However, with access to more computational resources and data, we would like to compare the results our present model produces to those from a model derived by fine-tuning or re-training.
  2. Implement a more advanced UI: As our project aims to help patients of MND, given more information about the hardware system used with our model, we would have liked to implement measures to make the UI as easy to use with minimal movement on the user's part. For instance, we could implement an eye-tracking system allow the user to navigate the system with their eyes. This will greatly benefit MND patients who have lost their motor functions as it allows them to still be able to use our application to generate emotional speech.
  3. Incorporating voice cloning: In future updates to this project, we would like to introduce voice cloning capabilities in addition to emotional inflections. This is possible by extending the frontend of our project to allow the user to record their own voice when setting up the software. This functionality will ensure that the user's own voice is utilised in the emotional audio.

About EMENDi2

EMENDi2 aims to significantly improve the communication process for MND patients by introducing an emotional voice synthesiser.
